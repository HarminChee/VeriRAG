Design for Testability (DFT) is a technique introduced during the design phase of integrated circuits (ICs) and electronic systems to enhance testability, reduce testing costs, and improve fault coverage and product quality. DFT enables efficient manufacturing testing, debugging, and fault diagnosis by ensuring that internal logic is controllable and observable during production testing, utilizing techniques such as scan chains (Scan Chain) and built-in self-test (BIST). An effective DFT design requires that all clock, reset, and data paths be compatible with scan testing, avoiding the use of internally generated clocks or uncontrollable asynchronous signals to ensure proper propagation and capture of test patterns, thereby improving test coverage and chip reliability.
The following are four common types of DFT errors. I also give detailed definitions for each. Please remember, understand and refer to them.
ACNCPI (Asynchronous Control Not Controllable from Primary Inputs) error occurs when the asynchronous control signal (such as asynchronous reset or set) of a flip-flop or latch is not directly controllable from the primary inputs. This issue prevents proper scan chain insertion, affecting DFT testability and fault coverage. In a well-structured design, asynchronous reset or set signals should be driven directly by primary input ports rather than being generated by internal signals or output ports. For example, in the given code snippet, the rst signal is declared as an output port but is also used as the reset signal for the register q, violating DFT design principles. The correct approach is to ensure that all asynchronous control signals originate from primary input ports, allowing direct control during testing, thus enabling effective scan testing and fault detection.
CLKNPI (Clock Not Derived from Primary Input) error occurs when a flip-flop is clocked by a signal that is not directly derived from a primary input. Internally generated clocks create testability challenges because flip-flops driven by such clocks cannot be included in the scan chain, reducing fault coverage in DFT. The RMM (Reuse Methodology Manual) section 5.4.4 also advises against using internally generated clocks for better testability. In the provided code example, the clock signal clk_a for flip-flop port_c is not driven from a primary input, leading to a violation. To resolve this issue, the design should be modified to ensure that all clock signals originate from primary inputs, thereby enabling effective scan chain insertion and improving test coverage.
FFCKNP (Flip-Flop Clock Not from Primary Input) error occurs when a flip-flop is clocked by a signal that is generated through another flip-flop. Internally generated clocks create testability challenges because flip-flops driven by such clocks cannot be included in the scan chain, reducing fault coverage in DFT. The RMM (Reuse Methodology Manual) section 5.4.4 also advises against using internally generated clocks for better testability. In the provided code example, the clock signal clk_a for flip-flop port_c is derived from another flip-flop instead of coming directly from a primary input, leading to a violation. To resolve this issue, the design should be modified so that all flip-flops receive their clock signals directly from a primary input, ensuring proper scan chain insertion and improving test coverage.
CDFDAT (Clock Driving Flip-Flop Data) error occurs when the clock signal is used as part of the data input for a flip-flop. This creates testability issues by making the data entering the flip-flop unobservable, leading to data capture problems and low fault coverage when performing scan-based testing. In the provided schematic, the clock (clk) is driving both the clock and data inputs of the same flip-flop, which can hinder scan chain insertion and reduce the controllability of the design. This issue should either be avoided or mitigated by isolating the clock signal in test mode using a dedicated test-mode signal. A similar check, CDLDAT, applies to latches.
